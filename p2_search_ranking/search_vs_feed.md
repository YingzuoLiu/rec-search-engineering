# Search vs Feed 推荐系统对比说明

**（Search、Feed 与对话式推荐的关系）**

本文用于说明 **搜索推荐（Search）** 与 **信息流推荐（Feed）** 在用户意图、建模方式和系统目标上的核心差异,并解释为什么项目二的建模方式更符合 Feed-style Ranking,而非传统搜索系统。

---

## 一、核心区别:用户意图的表达方式不同

### 1. Search(搜索推荐)

Search 场景中,用户意图是**显式表达**的:

- 用户直接输入 query
- 系统的任务是 **query → item / document** 的匹配

**典型输入示例:**

- "寿司 附近"
- "无线降噪耳机"
- "退款规则"

**主要信号包括:**

- Query 文本
- Query embedding
- Query–Item / Query–Document 相关度

**本质问题:**  
用户已经知道自己要什么,系统负责"找出来"。

### 2. Feed(信息流推荐)

Feed 场景中,用户意图是**隐式的**:

- 没有明确 query
- 系统需要从行为中推断用户当前想要什么

**常见信号包括:**

- 点击 / 浏览 / 购买序列
- 停留时长
- 行为的顺序与新近性(recency)

**典型场景:**

- 首页商品流
- 内容推荐流
- 无搜索入口的"发现页"

**本质问题:**  
用户没有明确说想要什么,系统要预测"下一步最可能感兴趣的内容"。

### 一句话总结:

> **Search 是回答「你问了什么」**  
> **Feed 是预测「你接下来可能想要什么」**

---

## 二、Query vs 行为:意图从哪里来?

| 维度 | Search | Feed |
|------|--------|------|
| 意图来源 | 显式 query | 隐式行为 |
| 意图稳定性 | 单次请求内相对稳定 | 高度动态 |
| 主要建模对象 | Query–Item 匹配 | 行为 → 意图 → Item |
| 系统目标 | 找最相关 | 预测最合适 |

**在项目二中:**

- 不存在显式 query
- 用户意图完全来自 **短期行为序列**
- 更符合 **Feed 推荐的建模假设**

---

## 三、为什么 Feed 更依赖序列建模

Feed 推荐通常具有以下特点:

- Session 较短
- 行为稀疏
- 用户意图变化快

**因此:**

- 新近行为(recency)非常重要
- 行为的顺序不能被忽略
- 简单地对历史行为做平均会丢失关键信息

**项目二的建模思路正是基于这一点:**

- 构造时间有序的用户 Session
- 使用 attention-style 方式聚合近期行为
- 明确聚焦 **短期意图建模**,而非长期偏好

---

## 四、为什么项目二不是 Search 系统

尽管 Search 与 Feed 在技术上有部分重叠(如 embedding、ranking 模型),但项目二不属于搜索系统,原因包括:

- 没有 query 解析或规范化
- 没有 lexical / semantic retrieval 阶段
- 没有 query–item 的匹配目标

**相反,项目二:**

- 基于最近行为预测下一个可能感兴趣的 item
- 本质是一个 **feed-style ranking 问题**
- 更接近电商或内容平台中的信息流推荐

---

## 五、Search 与 Feed 的技术重叠(但角色不同)

| 技术 | Search 中的角色 | Feed 中的角色 |
|------|----------------|---------------|
| Embedding | 表示 query 与 item | 表示用户状态与 item |
| Ranking | Query 条件下排序 | 行为条件下排序 |
| Attention | Query 对关键词聚焦 | 行为对 item 的加权 |
| Evaluation | 相关性 | 参与度 / 点击预测 |

**本项目刻意采用 Feed 视角 来理解这些技术。**

---

## 六、对话式推荐系统的定位

并非所有对话机器人都属于 Search。

- **问答型 / FAQ / 文档查询机器人** → 本质是 Search(只是交互形式是对话)
- **推荐 / 引导 / 转化型对话机器人** → 更接近 Feed 推荐

**在推荐型对话系统中:**

- 用户的每一句话都可视为一种行为信号
- 意图往往是逐步显现的
- 系统需要持续预测 "next best suggestion"

**因此:**

> 推荐型对话系统可以被视为 **Feed 推荐在对话场景下的延伸**。

---

## 七、系统设计层面的影响

Feed 推荐系统通常更加关注:

- 推理延迟
- 模型复杂度可控
- 对噪声行为的鲁棒性
- 对新近行为的快速响应

**项目二在设计中体现了这些取舍:**

- 使用轻量模型结构
- 优先建模短期序列
- 讨论效果与延迟之间的权衡

---

## 总结

本文从用户意图、信号来源、建模方式和系统目标等多个维度,系统性地区分了 Search 与 Feed 推荐系统的本质差异。项目二作为一个基于用户行为序列的推荐系统,在设计理念、技术选型和评估标准上,都更符合 Feed-style Ranking 的特征,而非传统的搜索匹配系统。
